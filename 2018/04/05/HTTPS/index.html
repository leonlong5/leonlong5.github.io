HTTPS

Posted on Apr 5 2018 | In Network

HTTP problems

  1. May be eavesdropped
    • Http does not have any encryption on data, HTTP packets are sent in clear text.
    • All data sent and received through internet may be intercepted or peeped.

  1. Certification issue

    • Unable to confirm that the server you are sending to is the real server(maybe the server is disguised)
    • Unable to confirm if returned client is a client(maybe the client is disguised) that is receiving as intended.
    • Unable to determine whether the other party that is communicating has access rights to some important information on web server. Also failed to prevent DoS(Denial of Service) attacks under massive requests

  2. May be intercepted

    • Data may be intercepted and Modified during the transmit between client and server(Man-in-the-Middle attack)

What is HTTPS

  1. HTTPS(Hyper Text Transfer Protocol over Secure Socket Layer), also known as HTTP over TLS/SSL. HTTPS is a transmission protocol for secure communication over internet. HTTP communicates via HTTP, but use SSL/TLS to encrypt packets. The main purpose of HTTPS development is to provide identity authentication for website servers and to protect the privacy and integrity of exchanged data.

  2. SSL/TLS

    • SSL/TLS use Public-key cryptography, that is, client send request for public key to server first, then use public key to encrypted message, after received this server use private key to decrypt the message. But this bring up two problems:

      1. How to prevent the public key to be intercepted?

        Put public key in public key certificate. As long as the certificate is certified, the public key is trusted.

      2. Asymmetric encryption involve a lot computation, how to reduce time used?

        Every session, client and server will generate a session key, use it to encrypt message. Because it is symmetrical encryption, and server public key only used to encrypt session key, so it is very fast.
        So the basic process of SSL/TLS is like this:

        1. Client send request to server, server put public key in certificate send back to client
        2. Client validate the certificate from authorized server, if the certificate is valid, client generate random number as session key, then encrypt it with public key send back to servers
        3. Server use private key to decrypt the message, get the session key, so now client and server can use the same session key to communication.
